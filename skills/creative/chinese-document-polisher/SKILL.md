# 中文文档润色增强

专业中文文档润色与改写工具。比 humanizer 更专注中文场景，比 prism 更关注表达而非逻辑审计。

## 何时使用

用户说以下任意一项时加载此 skill：
- "润色一下" / "帮我润色" / "polish this"
- "改写一下" / "重写" / "rewrite"
- "去AI味" / "太机械了" / "不像人写的"
- "缩写" / "精简" / "缩短"
- "扩写" / "展开" / "丰富"
- "改成正式一点" / "改成口语化" / "改成汇报风格"
- "邮件润色" / "公文润色" / "公众号润色"
- "帮我看看这段文字怎么改更好"

## 润色风格

| 风格关键词 | 适用场景 | 特点 |
|---|---|---|
| `简洁专业` | 技术文档、方案、spec | 短句、术语精准、无废话、逻辑链清晰 |
| `老板汇报` | 向上汇报、周报、月报 | 结论先行、数据支撑、要点分明、不超过3层 |
| `公众号` | 微信公众号、自媒体 | 标题感、节奏感、金句、适当口语化、段落短 |
| `技术白皮书` | 白皮书、架构文档、技术方案 | 严谨、有据可查、图表引用、术语统一 |
| `微信口语化` | 微信消息、群通知、私聊 | 短、自然、像真人说话、少书面语 |
| `正式邮件` | 商务邮件、对外函件 | 称谓规范、礼貌但不过度、重点突出 |
| `新闻通稿` | PR、公告、新闻 | 倒金字塔、5W1H、客观陈述、无主观判断 |
| `学术报告` | 论文、研究报告 | 严谨、引用规范、被动语态适度、术语准确 |

## 润色流程

### 第一步：判断需求

1. 用户是否指定了风格？没有则根据内容类型推断：
   - 邮件 → `正式邮件`
   - 微信/消息 → `微信口语化`
   - 含"汇报""报告" → `老板汇报`
   - 含"方案""设计""架构" → `简洁专业` 或 `技术白皮书`
   - 含"公众号""推文" → `公众号`
   - 学术内容 → `学术报告`
   - 无法判断 → 默认 `简洁专业`，并在说明中告知用户可换风格

2. 用户是润色、缩写、还是扩写？
   - 润色：保持长度，优化表达
   - 缩写：压缩到指定比例（默认压缩50%）
   - 扩写：丰富细节，增加例证

### 第二步：执行润色

核心原则：

1. **保留原意** — 不改变作者要传达的信息和立场
2. **去AI味** — 消除以下高频AI痕迹（完整特征库见 `references/ai-isms-checklist.md`）：
   - "首先…其次…最后…" 三段论
   - "值得注意的是" / "需要指出的是"
   - "总而言之" / "综上所述"（除非正式文档）
   - 过度对称的排比句
   - 每段都是"主题句+展开+总结"的模板
   - "在…方面" / "通过…的方式" 等翻译腔
   - "这不仅仅是…更是…" 递进煽情
   - 过多副词修饰："非常""极其""十分"
3. **提升可读性** — 短句优先、主动语态优先、具体优于抽象
4. **术语一致** — 同一文档中同一概念用同一个词
5. **逻辑连贯** — 句间衔接自然，不用生硬的"另外""此外"

### 第三步：输出格式

始终输出两部分：

```
## 润色结果

[润色后的完整文本]

## 修改说明

- **风格**：[使用的风格]
- **主要改动**：
  1. [具体改动1，如：去掉"首先其次最后"三段论，改为自然过渡]
  2. [具体改动2]
  3. [具体改动3]
- **保留**：[未改动的内容及原因]
- **建议**：[可选，如建议补充数据、建议拆分句子等]
```

## 特殊模式

### 批量润色
用户提供多段文字时，逐段润色，统一风格，最后给一个整体修改说明。

### 对比模式
用户说"对比"或"给我看改了哪里"时，用 diff 格式输出：

```
原文：我们需要在接下来的工作中重点关注以下几个方面
改后：接下来重点抓这几件事
```

### 多风格版本
用户说"给我几个版本"时，输出2-3个不同风格版本供选择。

### 保留术语表
技术文档润色时，先列出识别到的术语表，确认不改动这些术语，再润色其余部分。

## 不做的事

- 不改变事实信息（数字、名称、日期、引用）
- 不添加原文没有的观点或论据
- 不删除作者的核心论点
- 不在润色结果中夹带评论，评论只放在"修改说明"
- 除非用户要求"扩写"，否则不大幅增加篇幅

## 会议纪要补充/重写模式

当用户提供一份会议纪要模板（可能是 .docx 附件）和一个飞书 Wiki 链接，要求"根据Wiki补充会议纪要的第X部分"时：

1. **提取Wiki内容**：用飞书 API `GET /docx/v1/documents/{obj_token}/raw_content` 获取原始文本（Wiki node → obj_token → raw_content，详见 `feishu-sheet-api` skill）
2. **读取原始文档**：用 `python-docx` 或 `read_file` 读取 .docx 附件内容
3. **识别模板残留**：原始文档可能包含不匹配的内容（如模板复制后未修改的段落），需要全部替换
4. **按会议纪要格式重写**：
   - 保持原有头部信息（会议时间、地点、参会人员、主题）
   - 用 Wiki 内容填充指定章节
   - 按内部正式纪要风格撰写，不要AI味
   - 加入用户指定的人事决策（如"宋总评审通过"）
5. **生成新文档**：用 `ocr-and-documents` 的 MD→DOCX 脚本输出 .docx

### 去AI味要点（会议纪要特化）

- 用"本次会议主要围绕…进行评审"代替"本次会议旨在探讨"
- 用"会议对…进行了讨论"代替"会议深入分析了"
- 用"经会议讨论，XX对…进行了评审，并同意通过"代替"经过充分讨论后一致认为"
- 不要用"首先""其次""最后"三段论
- 参会人员名单、时间、地点等事实信息不可改动
